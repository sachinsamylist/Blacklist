<h2><a id="user-content-extensions" class="anchor" aria-hidden="true" href="#extensions"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Extensions</h2>
<p>The unified hosts file is extensible.  Extensions are used to block domains by category.</p>
<p>Extensions are optional, and are added to the base hosts file.  Extensions are combined in various ways wth the default hosts file, and the combined products are stored in the <a href="https://github.com/StevenBlack/hosts/tree/master/alternates"><code>alternates</code></a> folder.</p>
<p>For example, you may want to block porn domains in addition to the adware and malware we block by default.  That hosts file is stored in the porn subfolder of the <a href="https://github.com/StevenBlack/hosts/tree/master/alternates"><code>alternates</code></a> folder.</p>
<p>Data for extensions is stored in the <a href="https://github.com/StevenBlack/hosts/tree/master/extensions"><code>extensions</code></a> folder. You manage extensions by curating the
<a href="https://github.com/StevenBlack/hosts/tree/master/extensions"><code>extensions</code></a>  folder tree where you will find the data for <code>fakenews</code>, <code>social</code>, <code>gambling</code>, and <code>porn</code> extension data that we maintain and provide for you.</p>
<h2><a id="user-content-generate-your-own-unified-hosts-file" class="anchor" aria-hidden="true" href="#generate-your-own-unified-hosts-file"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Generate your own unified hosts file</h2>
<p>To generate your own unified hosts file you will need Python 3.5 or later.</p>
<p>First install the dependencies with:</p>
<pre><code>pip3 install --user -r requirements.txt
</code></pre>
<p><strong>Note</strong> we recommend the <code>--user</code> flag which installs the required dependencies at the user level. More information about it can be found on pip <a href="https://pip.pypa.io/en/stable/reference/pip_install/?highlight=--user#cmdoption-user" rel="nofollow">documentation</a>.</p>
<p>To run unit tests, in the top level directory, just run:</p>
<pre><code>python3 testUpdateHostsFile.py
</code></pre>
<p>The <code>updateHostsFile.py</code> script will generate a unified hosts file based on the sources in the
local <code>data/</code> subfolder.  The script will prompt you whether it should fetch updated versions
(from locations defined by the <code>update.json</code> text file in each source's folder). Otherwise, it
will use the <code>hosts</code> file that's already there.</p>

<h4><a id="user-content-command-line-options" class="anchor" aria-hidden="true" href="#command-line-options"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Command line options:</h4>
<p><code>--help</code>, or <code>-h</code>: display help.</p>
<p><code>--auto</code>, or <code>-a</code>: run the script without prompting. When <code>--auto</code> is invoked,</p>
<ul>
<li>Hosts data sources, including extensions, are updated.</li>
<li>No extensions are included by default.  Use the <code>--extensions</code> or <code>-e</code> flag
to include any you want.</li>
<li>Your active hosts file is <em>not</em> replaced unless you include the <code>--replace</code>
flag.</li>
</ul>
<p><code>--backup</code>, or <code>-b</code>: Make a backup of existing hosts file(s) as you generate
over them.</p>
<p><code>--extensions &lt;ext1&gt; &lt;ext2&gt; &lt;ext3&gt;</code>, or <code>-e &lt;ext1&gt; &lt;ext2&gt; &lt;ext3&gt;</code>: the names
of subfolders below the <code>extensions</code> folder containing additional
category-specific hosts files to include in the amalgamation. Example:
<code>--extensions porn</code> or <code>-e social porn</code>.</p>
<p><code>--flush-dns-cache</code>, or <code>-f</code>: skip the prompt for flushing the DNS cache.
Only active when <code>--replace</code> is also active.</p>
<p><code>--ip nnn.nnn.nnn.nnn</code>, or <code>-i nnn.nnn.nnn.nnn</code>: the IP address to use as the
target.  Default is <code>0.0.0.0</code>.</p>
<p><code>--keepdomaincomments</code>, or <code>-k</code>: <code>true</code> (default) or <code>false</code>, keep the comments
that appear on the same line as domains.  The default is <code>true</code>.</p>
<p><code>--noupdate</code>, or <code>-n</code>: skip fetching updates from hosts data sources.</p>
<p><code>--output &lt;subfolder&gt;</code>, or <code>-o &lt;subfolder&gt;</code>: place the generated source file
in a subfolder.  If the subfolder does not exist, it will be created.</p>
<p><code>--replace</code>, or <code>-r</code>: trigger replacing your active hosts</p>
<p><code>--skipstatichosts</code>, or <code>-s</code>: <code>false</code> (default) or <code>true</code>, omit the standard
section at the top, containing lines like <code>127.0.0.1 localhost</code>.  This is
useful for configuring proximate DNS services on the local network.</p>
<p><code>--compress</code>, or <code>-c</code>: <code>false</code> (default) or <code>true</code>, <em>Compress</em> the hosts file
ignoring non-necessary lines (empty lines and comments) and putting multiple
domains in each line. Reducing the number of lines of the hosts file improves
the performances under Windows (with DNS Client service enabled).</p>
<p><code>--minimise</code>, or <code>-m</code>: <code>false</code> (default) or <code>true</code>, like <code>--compress</code>, but puts
each domain on a separate line. This is necessary because many implementations
of URL blockers that rely on <code>hosts</code> files do not conform to the standard which
allows multiple hosts on a single line.</p>
<h2><a id="user-content-how-do-i-control-which-sources-are-unified" class="anchor" aria-hidden="true" href="#how-do-i-control-which-sources-are-unified"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>How do I control which sources are unified?</h2>
<p>Add one or more  <em>additional</em> sources, each in a subfolder of the <code>data/</code>
folder, and specify the <code>url</code> key in its <code>update.json</code> file.</p>
<p>Add one or more <em>optional</em> extensions, which originate from subfolders of the
<code>extensions/</code> folder.  Again the url in <code>update.json</code> controls where this
extension finds its updates.</p>
<p>Create an <em>optional</em> <code>blacklist</code> file. The contents of this file (containing a
listing of additional domains in <code>hosts</code> file format) are appended to the
unified hosts file during the update process. A sample <code>blacklist</code> is
included, and may be modified as you desire.</p>
<ul>
<li>NOTE: The <code>blacklist</code> is not tracked by git, so any changes you make won't
be overridden when you <code>git pull</code>   this repo from <code>origin</code> in the future.</li>
</ul>

<p>If you have custom hosts records, place them in file <code>myhosts</code>.  The contents
of this file are prepended to the unified hosts file during the update
process.</p>
<p>The <code>myhosts</code> file is not tracked by git, so any changes you make won't be
overridden when you <code>git pull</code> this repo from <code>origin</code> in the future.</p>
<h3><a id="user-content-how-do-i-prevent-domains-from-being-included" class="anchor" aria-hidden="true" href="#how-do-i-prevent-domains-from-being-included"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>How do I prevent domains from being included?</h3>
<p>The domains you list in the <code>whitelist</code> file are excluded from the final hosts
file.</p>
<p>The <code>whitelist</code> uses partial matching.  Therefore if you whitelist
<code>google-analytics.com</code>, that domain and all its subdomains won't be merged
into the final hosts file.</p>
<p>The <code>whitelist</code> is not tracked by git, so any changes you make won't be
overridden when you <code>git pull</code> this repo  from <code>origin</code> in the future.</p>

<h2><a id="user-content-what-is-a-hosts-file" class="anchor" aria-hidden="true" href="#what-is-a-hosts-file"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>What is a hosts file?</h2>
<p>A hosts file, named <code>hosts</code> (with no file extension), is a plain-text file
used by all operating systems to map hostnames to IP addresses.</p>
<p>In most operating systems, the <code>hosts</code> file is preferential to <code>DNS</code>.
Therefore if a domain name is resolved by the <code>hosts</code> file, the request never
leaves your computer.</p>
<p>Having a smart <code>hosts</code> file goes a long way towards blocking malware, adware,
and other irritants.</p>
<p>For example, to nullify requests to some doubleclick.net servers, adding these
lines to your hosts file will do it:</p>
<pre><code># block doubleClick's servers
0.0.0.0 ad.ae.doubleclick.net
0.0.0.0 ad.ar.doubleclick.net
0.0.0.0 ad.at.doubleclick.net
0.0.0.0 ad.au.doubleclick.net
0.0.0.0 ad.be.doubleclick.net
# etc...
</code></pre>
<h2><a id="user-content-we-recommend-using-0000-instead-of-127001" class="anchor" aria-hidden="true" href="#we-recommend-using-0000-instead-of-127001"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>We recommend using <code>0.0.0.0</code> instead of <code>127.0.0.1</code></h2>
<p>Traditionally most host files use <code>127.0.0.1</code>, the <em>loopback address</em>, to establish an IP connection to the local machine.</p>
<p>We prefer to use <code>0.0.0.0</code>, which is defined as a non-routable meta-address used to designate an invalid, unknown,
or non applicable target.</p>
<p>Using <code>0.0.0.0</code> is empirically faster, possibly because there's no wait for a timeout resolution. It also does not
interfere with a web server that may be running on the local PC.</p>
<h2><a id="user-content-why-not-use-just-0-instead-of-0000" class="anchor" aria-hidden="true" href="#why-not-use-just-0-instead-of-0000"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Why not use just <code>0</code> instead of <code>0.0.0.0</code>?</h2>
<p>We tried that.  Using <code>0</code> doesn't work universally.</p>
<h2><a id="user-content-location-of-your-hosts-file" class="anchor" aria-hidden="true" href="#location-of-your-hosts-file"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Location of your hosts file</h2>
<p>To modify your current <code>hosts</code> file, look for it in the following places and modify it with a text

<p><strong>Mac OS X, iOS, Android, Linux</strong>: <code>/etc/hosts</code> file.</p>
<p><strong>Windows</strong>: <code>%SystemRoot%\system32\drivers\etc\hosts</code> file.</p>

<h2><a id="user-content-goals-of-this-unified-hosts-file" class="anchor" aria-hidden="true" href="#goals-of-this-unified-hosts-file"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Goals of this unified hosts file</h2>
<p>The goals of this repo are to:</p>
<ol>
<li>
<p>automatically combine high-quality lists of hosts,</p>
</li>
<li>
<p>provide easy extensions,</p>
</li>
<li>
<p>de-dupe the resultant combined list,</p>
</li>
<li>
<p>and keep the resultant file reasonably sized.</p>
</li>
</ol>
<p>A high-quality source is defined here as one that is actively curated.  A
hosts source should be frequently updated by its maintainers with both
additions and removals.  The larger the hosts file, the higher the level of
curation is expected.</p>
<p>For example, the (huge) hosts file from <a href="https://hosts-file.net" rel="nofollow">hosts-file.net</a>
is <strong>not</strong> included here because it is very large (780,000+ entries)
and doesn't currently display a corresponding high level of curation activity.</p>
<p>It is expected that this unified hosts file will serve both desktop and mobile
devices under a variety of operating systems.</p>
